Git - A Version Control System

Have you ever created a file and renamed it agian and agian as you made changes to it. Like file.txt > file1.txt > file2.txt > Final_file.txt and so onn... boring isnt it?
To handle this thing we have a tool called Git. It manages the version of files by creating snapshot(or picture of files) and store the refrence of them and hence called the Version Control System(VCS). Git can be downloded here.
There are 3 states for git, where your files can reside- modified, staged, and committed

  • Modified - Changed the file but have not staged.
  • Staged - You have marked modified file in its current version to go into your next commit snapshot.
  • Commit - The data is safely stored in your local database.

Hence we have the main 3 sections of git -- working tree, staging area & git directory

  • Working Directory - It reside on you local machine where you make changes to the files.
  • Staging Area - A file in .git directory containing info about what will go into your next commit.
  • .git Directory - stores the metadata and object database for your project.


Now let's move on to some practical world. We will create a small file and modify it.

  • Lets create a directory and change it first on local machine using $ mkdir Pi; cd Pi
  • Initialize this dir so that git can track files (This will create a .git directory in your project) $ git init .
  • Create a file and save it
  • Use $ git status to check what changes are made
  • After creating the file the status will be untracked. To track the file use $ git add <filename>
  • After adding the the files is in staging area need to commit them i.e give a message what & why it was changed.
    $git commit -m "First commit"
  • If any changes are made go up 3 lines and start again
